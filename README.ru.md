Русская документация переведена с помощью ИИ.

Я сделал этот MCP-сервер для управления обычным хостингом Beget из разных клиентов. В примерах встречается Codex, но сам сервер от него не зависит. Он так же работает с JetBrains AI Assistant, Claude Desktop, Cursor, VS Code и другими совместимыми клиентами.

Сервер написан на Go, а весь граф зависимостей собран на Uber Fx. `app.Run` выбирает запуск MCP или команды управления credentials, при этом оба пути получают зависимости через Fx. Конфигурация, постоянное хранилище credentials, HTTP-клиент, адаптер Beget, MCP и жизненный цикл процесса оформлены отдельными модулями.

## Что уже можно делать

Без изменения аккаунта:

- получить сведения об аккаунте, DNS, доменах, FTP, MySQL, сайтах, Cron, бэкапах, почте и нагрузке
- например, посмотреть почтовые ящики, FTP-аккаунты, базы MySQL, файловые бэкапы и задачи Cron

С изменением аккаунта:

- выполнить все опубликованные операции этих разделов, включая смену пароля почтового ящика, создание ящиков и переадресаций, смену FTP- и MySQL-паролей, управление сайтами, доменами, бэкапами и Cron

Всего сервер предоставляет 66 типизированных инструментов. Каждый инструмент жёстко привязан к одному опубликованному методу Beget и его документированным параметрам: универсального прокси для произвольных вызовов API здесь нет.

Каждая входная схема содержит только параметры своего метода Beget. Обязательные поля, идентификаторы, enum-значения, диапазоны, пути, Cron-выражения и несовместимые значения проверяются до HTTP-запроса. Поля паролей управляемых FTP-, MySQL- и почтовых ресурсов помечены write-only и не попадают в итоговые описания.

Каждый инструмент возвращает единый типизированный envelope с полями `success`, `result` и `errors`. Read-only операции отдают документированные поля Beget через модель конкретной операции. Результат изменения содержит `changed` и типизированные данные провайдера, включая `changed: false`, если запрос отклонён или его исход неизвестен. Ошибки имеют стабильные категории для валидации, авторизации, отказа провайдера, транспортного сбоя, отсутствия подтверждения и неизвестного исхода mutation, а также содержат безопасный следующий шаг. Недокументированные поля провайдера сохраняются в `additional_properties_json` как точные JSON-значения, а пустые списки всегда возвращаются как `[]`.

Пароль почтового ящика содержит от 6 до 64 символов. Допустимы только английские буквы, цифры и ``.,/<>?;:"'`!@#$%^&*()[]{}_+-=|~``. Обязательны хотя бы одна буква, одна цифра и один знак. Сервер проверяет политику локально и безопасно объясняет ошибку Beget 1208, не повторяя пароль.

Любой MCP-клиент получает при инициализации короткий безопасный порядок работы: проверить авторизацию, если её состояние неизвестно, прочитать текущее состояние и реальные идентификаторы перед изменением, получить явное подтверждение, выполнить одно подтверждённое изменение и проверить результат. Устанавливаемый skill для Codex дополняет этот порядок выбором инструментов и обработкой ошибок, не дублируя схемы.

У изменяющих инструментов есть два уровня защиты. MCP-клиент видит точные destructive и idempotent annotations, а сервер требует явно передать `confirm: true`. Без подтверждения запрос к Beget не выполняется.

Я не добавлял универсальный инструмент для произвольных методов API. Такой инструмент проще написать, но им намного легче случайно удалить или изменить что-нибудь важное.

## Версии и зависимости

Проект рассчитан на Go 1.26 и фиксирует toolchain `go1.26.5`.

Основные зависимости:

- Uber Fx 1.24.0 отвечает за сборку приложения
- Testify 1.11.1 используется во всех тестах через `assert` и `require`
- `basic-utils/v3` 3.4.0 помогает аккуратно читать настройки окружения
- `go-keyring` 0.2.8 переносит credentials, сохраненные предыдущими выпусками
- официальный Go SDK реализует протокол MCP

## Транспорты

Сервер поддерживает три взаимоисключающих транспорта:

- stdio используется по умолчанию и явно включается через `--stdio`
- Streamable HTTP включается через `--streamable-http`
- старый SSE включается через `--sse`

HTTP-транспорты по умолчанию слушают `127.0.0.1:8080` и не могут привязаться к внешнему адресу. Для endpoint, поведения сессий, формата ответа и опциональной bearer-авторизации предусмотрены отдельные флаги.

Все флаги и примеры настройки клиентов собраны в [инструкции по транспортам](docs/transports.ru.md).

## Справка командной строки

Запустите бинарник с `help`, `-h` или `--help`, чтобы увидеть команды и параметры транспортов:

```bash
beget-api-mcp-server help
```

Справка по отдельным командам доступна без запуска MCP-сервера и обращения к системному хранилищу:

```bash
beget-api-mcp-server help credentials
beget-api-mcp-server help credentials set
beget-api-mcp-server help upgrade
```

## Сборка

```bash
go build -o bin/beget-api-mcp-server ./cmd/beget-api-mcp-server
```

Проверки перед коммитом:

```bash
go fmt ./...
go vet ./...
go test -race ./...
```

Те же проверки запускаются в GitHub Actions. Отчет покрытия сохраняется как артефакт сборки.

Полный набор тестов, проверки покрытия, линтеров, поиска уязвимостей, статического анализа безопасности и поиска секретов запускается командой `task verify`. Перед первым запуском нужно установить зафиксированные версии инструментов через `task tools`. Обязательный порог покрытия составляет 90%, текущий набор тестов покрывает 91,1%, а badge публикуется из ветки `badges`. Те же категории проверок выполняются в GitHub Actions, а Dependabot следит за модулями Go и workflow.

Команда `task benchmark` измеряет запуск сервера, инициализацию MCP, `tools/list`, генерацию схем, вызовы инструментов и HTTP round trip через локальный fake server. Она выводит время, байты и allocations без обращения к Beget. Текущий эталонный замер приведен в [документе о быстродействии](docs/performance.ru.md).

Команда `task mcp-inspector` запускает зафиксированную версию официального MCP Inspector для интерактивной проверки протокола и инструментов. Для нее нужны Node.js, npm и доступная команда `npx`.

## Установка в систему

Последний выпуск можно глобально установить для текущего пользователя Linux или macOS одной командой:

```bash
curl -fsSL https://raw.githubusercontent.com/kordax/beget-api-mcp-server/main/install.sh | sh
```

Для Windows PowerShell:

```powershell
irm https://raw.githubusercontent.com/kordax/beget-api-mcp-server/main/install.ps1 | iex
```

Установщик сам определяет систему и архитектуру, проверяет контрольную сумму выпуска и добавляет команду в пользовательский `PATH`. Ручная установка, обновление, удаление и настройка MCP-клиентов описаны в [отдельной инструкции](docs/installation.ru.md).

Установленный бинарник умеет обновлять себя:

```bash
beget-api-mcp-server upgrade
```

Команда `upgrade --check` только сравнивает версии, а конкретный выпуск можно указать как `upgrade v0.3.0`. В интерактивном терминале во время проверки и установки показывается анимированный индикатор. Перед атомарной заменой исполняемого файла проверяется опубликованная SHA-256.

Работающий MCP-сервер также запускает фоновую проверку выпуска при первом вызове инструмента после десяти минут без вызовов. Проверка не задерживает операцию Beget. Если доступна новая версия, уведомление с установленной и доступной версиями добавляется в первый ответ инструмента, завершённый после проверки. Сервер никогда не устанавливает выпуск автоматически, а ошибка проверки не прерывает вызов инструмента Beget.

## Настройка доступа

Сначала в панели Beget нужно включить доступ к Hosting API и задать отдельный пароль для API. Обычный пароль от панели лучше не использовать.

Учетные данные можно один раз сохранить в постоянном хранилище сервера:

```bash
beget-api-mcp-server credentials set --login your-login
```

Команда скрыто запрашивает API-ключ в терминале. Для автоматизации ключ можно передать через stdin. Credentials записываются в стандартный пользовательский каталог конфигурации. В Unix-системах сервер устанавливает права `0700` на свой каталог и отклоняет файлы, доступные группе или другим пользователям; сам файл создается с правами `0600`. В Windows используется каталог AppData текущего пользователя и унаследованный пользовательский ACL. Проверить наличие или удалить данные можно без вывода их значений:

```bash
beget-api-mcp-server credentials check
beget-api-mcp-server credentials delete
```

После этого MCP-клиенту достаточно указать команду:

```toml
[mcp_servers.beget]
command = "beget-api-mcp-server"
```

Stdio является транспортом по умолчанию, поэтому указывать его аргументом не требуется.

Каждый процесс сервера читает один постоянный файл, поэтому credentials сохраняются между перезапусками и независимыми MCP-процессами. Записи Secret Service, Keychain или Credential Manager, созданные предыдущими выпусками, автоматически переносятся при первом чтении. MCP-сервер также запускается без настроенных учетных данных: агент может вызвать `beget_auth_status` и получить безопасную инструкцию, а сервер повторяет чтение постоянного хранилища, пока credentials отсутствуют. После успешной загрузки они дополнительно кэшируются в памяти до остановки процесса. Универсальные правила безопасности передаются через инициализацию MCP. Установщик также добавляет Codex skill `beget-api` с дополнительными правилами выбора инструментов и обработки ошибок.

Переменные `BEGET_API_LOGIN` и `BEGET_API_KEY` продолжают поддерживаться и имеют приоритет над сохраненными значениями. Они подходят для контейнеров, CI и запуска через внешний менеджер паролей.

Ключ уходит в Beget внутри HTTPS POST-запроса. Он не попадает в URL, логи, аргументы MCP-инструментов или ответы. Тесты работают только с локальным HTTP-сервером и вымышленными данными.

## Границы проекта

Проект намеренно поддерживает классический Hosting API по адресу `https://api.beget.com/api`. Расширение или изменение этого внешнего API не входит в область репозитория: сервер остается небольшим типизированным адаптером над поддерживаемыми операциями.
