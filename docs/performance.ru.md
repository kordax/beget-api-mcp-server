# Эталон быстродействия

Полный локальный набор benchmark-тестов запускается командой:

```bash
task benchmark
```

Набор использует MCP-транспорты в памяти, вымышленные credentials и локальный тестовый HTTP-сервер. Обращений к Beget нет. Каждый benchmark выводит задержку, объём выделенной памяти и количество allocations. Benchmark-тесты MCP-каталогов также показывают сериализованный размер ответа, а `BenchmarkMCPToolsList` выводит число инструментов.

## Эталонный запуск

Этот диагностический baseline получен 2026-07-13 с Go 1.26.5 на Linux amd64 и AMD Ryzen 9 9950X3D. Значения не являются порогами CI, поскольку планировщик и различия оборудования создают нормальные колебания.

| Benchmark | Время операции | Байт на операцию | Allocations |
| --- | ---: | ---: | ---: |
| Запуск сервера | 14,14 мс | 8 384 961 | 229 131 |
| Инициализация MCP | 123,3 мкс | 309 996 | 216 |
| MCP `tools/list` | 11,19 мс | 7 563 809 | 48 158 |
| MCP capability resource | 121,6 мкс | 228 727 | 109 |
| Самая большая входная схема Cron | 7,715 мкс | 15 072 | 52 |
| Вложенная схема директив | 7,050 мкс | 14 976 | 60 |
| Вызов MCP-инструмента | 96,50 мкс | 340 272 | 256 |
| Конкурентные вызовы MCP-инструмента | 60,36 мкс | 340 332 | 258 |
| Локальный HTTP round trip | 26,54 мкс | 10 246 | 116 |
| Конкурентные локальные HTTP round trip | 35,92 мкс | 38 120 | 147 |

Ответ `tools/list` содержит все 67 инструментов и занимает 167 349 сериализованных байт с отдельными входными и выходными схемами операций и контрактом результата dry-run для каждой mutation. Опциональный capability resource занимает 4 759 сериализованных байт, примерно в 35 раз меньше, и читается только при неясном выборе маршрута. Таблица нужна для сравнения результатов профилирования и поиска регрессий, а не как обещание одинаковых значений на другом компьютере.

В CI пока намеренно нет порога по latency или allocations. Эти значения остаются диагностическими, пока повторные запуски на стабильных runner-окружениях не покажут нормальный разброс и обоснованный допуск. Детерминированные проверки числа инструментов и сериализованного размера каталогов остаются обычными тестами; временной performance gate стоит добавлять только после подтверждения стабильности baseline.
