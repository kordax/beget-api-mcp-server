# Установка в систему

Эта инструкция устанавливает сервер один раз для текущего пользователя и подключает его к MCP-клиенту. Codex здесь является одним из примеров. В основной настройке используется stdio. Streamable HTTP и старый SSE описаны в [инструкции по транспортам](transports.ru.md).

## Что понадобится

Готовые архивы не требуют установленного Go. На странице [GitHub Releases](https://github.com/kordax/beget-api-mcp-server/releases) нужно выбрать Linux, macOS или Windows и архитектуру `amd64` или `arm64`. Каждый выпуск содержит `checksums.txt` для проверки SHA-256.

Для установки из исходников нужен Go 1.26.5:

```bash
go version
```

## Первая установка

Последний выпуск можно глобально установить для текущего пользователя Linux или macOS одной командой:

```bash
curl -fsSL https://raw.githubusercontent.com/kordax/beget-api-mcp-server/main/install.sh | sh
```

Для Windows PowerShell:

```powershell
irm https://raw.githubusercontent.com/kordax/beget-api-mcp-server/main/install.ps1 | iex
```

Установщик сам определяет систему и архитектуру, скачивает подходящий выпуск, проверяет SHA-256 и добавляет `beget-api-mcp-server` в пользовательский `PATH`. `sudo` скрыто не вызывается. После первой установки нужно перезапустить терминал и проверить команду:

```bash
command -v beget-api-mcp-server
```

Переменная `BEGET_MCP_VERSION` позволяет выбрать конкретную версию, а `BEGET_MCP_INSTALL_DIR` задает другой каталог. Для ручной установки на странице выпусков остаются готовые архивы и контрольные суммы.

Вместе с выпуском установщик добавляет skill `beget-api` для Codex. Каждый MCP-клиент получает критичные правила авторизации и изменений во время инициализации MCP и может вызвать `beget_auth_status` для актуальной инструкции по настройке.

## Сохранение учетных данных Beget

В панели Beget нужно включить Hosting API и создать отдельный пароль API. Логин и пароль сохраняются в постоянном хранилище сервера:

```bash
beget-api-mcp-server credentials set --login your-beget-login
```

API-ключ вводится через скрытый запрос терминала и не принимается в аргументах команды. Команда записывает versioned-файл в стандартный пользовательский каталог конфигурации. В Linux это обычно `~/.config/beget-api-mcp-server/credentials.json`, в macOS каталог находится внутри `~/Library/Application Support`, а в Windows используется AppData текущего пользователя.

Проверить наличие или удалить сохраненные данные можно без вывода их значений:

```bash
beget-api-mcp-server credentials check
beget-api-mcp-server credentials delete
```

В Unix-системах каталог получает права `0700`, а файл `0600`. Сервер отказывается читать файл или каталог, доступный группе или другим пользователям. В Windows файл наследует ACL пользовательского каталога AppData. Credentials, созданные предыдущим выпуском в Secret Service, Keychain или Credential Manager, автоматически переносятся при первом чтении.

Для запуска MCP учетные данные необязательны. Без них сервер все равно подключается к клиенту и предоставляет `beget_auth_status`. Реальная операция Beget будет возвращать короткую ошибку авторизации до настройки credentials. Работающий сервер без авторизации повторно читает постоянное хранилище при следующем запросе, поэтому после `credentials set` переподключение не требуется.

## Универсальный контракт MCP

По умолчанию сервер работает через stdio и загружает данные из постоянного пользовательского хранилища. Он не обращается к API Codex и не требует специальной обертки запуска. Обычный MCP-клиент с JSON-конфигурацией может запускать его так:

```json
{
  "mcpServers": {
    "beget": {
      "command": "beget-api-mcp-server"
    }
  }
}
```

Stdio используется по умолчанию, поэтому в конфигурации не нужны ни абсолютный путь, ни аргументы транспорта. Такой формат подходит клиентам с распространенной JSON-структурой `mcpServers`.

## Пример для Codex

Codex использует TOML вместо JSON-структуры выше. В `~/.codex/config.toml` можно добавить:

```toml
[mcp_servers.beget]
command = "beget-api-mcp-server"
```

После изменения конфигурации Codex нужно перезапустить. Это только представление того же универсального контракта команды и переменных окружения в формате Codex.

## Пример для JetBrains и GoLand

GoLand и другие актуальные JetBrains IDE умеют запускать локальные MCP-серверы через stdio. Нужно открыть `Settings | Tools | AI Assistant | Model Context Protocol (MCP)`, нажать `Add`, выбрать JSON-конфигурацию для STDIO и установить уровень сервера `Global`.

Сервер Beget можно добавить такой JSON-конфигурацией:

```json
{
  "mcpServers": {
    "beget": {
      "command": "beget-api-mcp-server"
    }
  }
}
```

После сохранения нужно нажать `OK`, затем `Apply`. В колонке статуса должно появиться успешное подключение, а в списке инструментов должны быть команды Beget. Если автоматический запуск отключен, сервер нужно включить вручную или нажать `Reconnect`.

Чтобы пользовательские MCP-серверы были доступны Junie, нужно открыть `Settings | Tools | AI Assistant | Agents` и включить `Pass custom MCP servers`.

Если процесс не запускается, сначала нужно перезапустить IDE, чтобы она получила обновленный пользовательский `PATH`. Затем можно открыть `Help | Show Log in Explorer`, перейти в каталог `mcp` и посмотреть лог сервера Beget.

## Хранение секрета

Для локальной установки рекомендуется встроенное постоянное хранилище: все процессы одного пользователя читают один защищенный файл. Если заданы переменные окружения, они имеют приоритет:

```bash
BEGET_API_LOGIN=your-beget-login \
BEGET_API_KEY=your-api-password \
beget-api-mcp-server --stdio
```

Этот fallback предназначен для контейнеров, CI и внешних менеджеров паролей. API-ключ нельзя помещать в аргументы процесса или коммитить в конфигурацию MCP.

Файл credentials записывается через закрытый временный файл, сбрасывается на диск перед заменой и никогда не попадает в результаты MCP или логи. После загрузки credentials дополнительно кэшируются в памяти до остановки процесса. Команда `credentials delete` удаляет постоянный файл и оставшиеся записи старого keyring.

## Установка из локального клона

Этот вариант удобен во время разработки сервера и требует Git:

```bash
git clone https://github.com/kordax/beget-api-mcp-server.git
cd beget-api-mcp-server
go install ./cmd/beget-api-mcp-server
```

Конфигурацию клиента менять не придется, потому что путь установленного бинарника останется тем же.

## Обновление

Для обновления используется встроенная команда:

```bash
beget-api-mcp-server upgrade
```

Можно только проверить версию или выбрать конкретный выпуск:

```bash
beget-api-mcp-server upgrade --check
beget-api-mcp-server upgrade v0.3.0
```

Команда определяет текущую платформу, скачивает подходящий выпуск, проверяет SHA-256 и атомарно заменяет исполняемый файл. В интерактивном терминале анимированный индикатор показывается во время поиска выпуска и установки обновления. При перенаправлении вывода и в логах CI управляющие последовательности терминала не появляются. На Windows предыдущий файл сохраняется до успешной замены. `GH_TOKEN` или `GITHUB_TOKEN` нужен только для приватного GitHub-репозитория, публичные выпуски скачиваются без токена. После обновления MCP-сервер нужно перезапустить или переподключить.

Во время работы MCP-сервер проверяет последний выпуск при первом вызове инструмента после десяти минут без вызовов. Доступная версия указывается в ответе инструмента, но никогда не устанавливается автоматически. Тайм-аут или ошибка сервиса выпусков записывается в лог и не меняет результат инструмента Beget.

Повторный запуск однострочного установщика остается запасным способом восстановления.

## Удаление

Нужно выполнить `beget-api-mcp-server credentials delete`, убрать сервер Beget из конфигурации MCP-клиента и удалить команду, путь к которой покажет `command -v beget-api-mcp-server`. Тот же путь установщик выводит после установки.

## Про безопасность

Ключ API отправляется в Beget только внутри HTTPS POST-запроса. Сервер не помещает его в URL, логи, аргументы MCP-инструментов или результаты. Способ передачи ключа процессу выбирает MCP-клиент или используемый менеджер секретов.
